# L2 - Detailed Requirements

## Toromont Fleet Hub — Equipment Fleet Management & Service Portal

---

## L2-001: Dashboard KPI Cards
**Traces to:** L1-001

The dashboard must display KPI cards showing: total equipment count, active equipment count, equipment requiring service, overdue work orders, and total fleet utilization percentage. Each KPI card must display the current value and a trend indicator (up/down/stable) compared to the previous 30-day period.

**Acceptance Criteria:**
1. Given an authenticated Fleet Manager user, when they navigate to the dashboard, then they must see KPI cards for total equipment, active equipment, service required, overdue work orders, and fleet utilization percentage.
2. Given equipment data exists for the current and previous 30-day periods, when the dashboard loads, then each KPI card must display a trend arrow (up, down, or stable) based on comparison with the prior period.
3. Given viewport width < 576px, when the user views the dashboard, then KPI cards must stack in a single column layout.
4. Given viewport width ≥ 992px, when the user views the dashboard, then KPI cards must display in a horizontal row.

---

## L2-002: Dashboard Equipment Map
**Traces to:** L1-001

The dashboard must display an interactive map showing the GPS locations of all equipment in the user's fleet. Equipment markers must be color-coded by status (operational=green, needs-service=yellow, critical=red, offline=gray). Clicking a marker must show a popup with equipment name, model, status, and a link to the equipment detail page.

**Acceptance Criteria:**
1. Given an authenticated user with equipment that has GPS coordinates, when the dashboard loads, then all equipment must appear as markers on the map at their last known positions.
2. Given equipment with status "operational", when displayed on the map, then the marker must be green.
3. Given equipment with status "critical", when displayed on the map, then the marker must be red.
4. Given a user clicks an equipment marker on the map, then a popup must display the equipment name, model, current status, and a clickable link to the equipment detail page.
5. Given viewport width < 768px, when the user views the dashboard, then the map must occupy full width and reduce to 250px height.

---

## L2-003: Dashboard Active Alerts Panel
**Traces to:** L1-001

The dashboard must display a panel listing the 10 most recent active alerts, ordered by severity (critical first) then by timestamp (newest first). Each alert must show the equipment name, alert type, severity level, and time since creation. The panel must include a "View All" link to the full alerts page.

**Acceptance Criteria:**
1. Given active alerts exist, when the dashboard loads, then the alerts panel must display up to 10 alerts ordered by severity (critical, warning, info) then by newest timestamp.
2. Given an alert for equipment "CAT 320 Excavator" with severity "critical", when displayed in the alerts panel, then it must show the equipment name, alert type, severity badge, and relative time (e.g., "2 hours ago").
3. Given more than 10 active alerts exist, when the dashboard loads, then only the top 10 must be shown with a "View All" link visible.

## L2-004: Equipment List View
**Traces to:** L1-002

The system must display a paginated, sortable, and filterable data grid of all equipment in the user's organization. The grid must include columns for: equipment ID, name, model, serial number, category, status, location, and last service date. Users must be able to filter by status, category, and location, and search by name or serial number.

**Acceptance Criteria:**
1. Given an authenticated user with 50 equipment records, when they navigate to the equipment list, then a Kendo UI data grid must display equipment with columns for ID, name, model, serial number, category, status, location, and last service date.
2. Given the equipment list is displayed, when the user clicks a column header, then the grid must sort by that column in ascending order; clicking again must sort descending.
3. Given the equipment list is displayed, when the user selects "Needs Service" from the status filter dropdown, then only equipment with that status must be shown.
4. Given the equipment list is displayed, when the user types "CAT 320" in the search box, then only equipment matching that name or serial number must appear.
5. Given more than 20 equipment records match the current filters, when the grid loads, then pagination controls must appear with 20 items per page.
6. Given viewport width < 768px, when the user views the equipment list, then the grid must switch to a card-based layout showing key fields only.

## L2-005: Equipment Detail View
**Traces to:** L1-002

The system must display a detailed view for a single piece of equipment, showing: specifications (make, model, year, serial number, weight, engine type), current status, location on a map, recent telemetry summary, service history timeline, and associated documents. The view must include action buttons for scheduling service and viewing full telemetry.

**Acceptance Criteria:**
1. Given an authenticated user, when they navigate to an equipment detail page, then they must see the equipment's full specifications, current status badge, and location displayed on a mini-map.
2. Given an equipment has service history records, when the detail page loads, then a timeline must display the 5 most recent service events with date, type, and summary.
3. Given an equipment has telemetry data, when the detail page loads, then a summary panel must show latest engine hours, fuel level, and operating temperature.
4. Given an authenticated Fleet Manager, when they view the equipment detail page, then "Schedule Service" and "View Telemetry" action buttons must be visible.
5. Given an authenticated Operator user, when they view the equipment detail page, then the "Schedule Service" button must not be visible (read-only access).

## L2-006: Register New Equipment
**Traces to:** L1-002

The system must provide a form to register new equipment with required fields: name, make, model, year, serial number, category, and initial status. Optional fields must include: GPS device ID, purchase date, warranty expiration, and notes. The form must validate all inputs and prevent duplicate serial numbers within the organization.

**Acceptance Criteria:**
1. Given an authenticated Admin or Fleet Manager, when they click "Add Equipment", then a registration form must appear with all required and optional fields.
2. Given valid equipment data is entered, when the user submits the form, then the equipment must be created and the user must be redirected to the new equipment's detail page.
3. Given a serial number that already exists in the organization, when the user submits the form, then a validation error must display: "Serial number already exists in your organization."
4. Given required fields are left empty, when the user submits the form, then inline validation errors must appear for each missing required field.
5. Given an authenticated Operator or Technician, when they attempt to access the Add Equipment form, then they must be denied access with a 403 Forbidden response.

## L2-007: Create Work Order
**Traces to:** L1-003

The system must allow Fleet Managers and Technicians to create work orders with: equipment selection, service type (preventive/corrective/emergency), priority (low/medium/high/critical), description, requested completion date, and assigned technician. Creating a work order must trigger a notification to the assigned technician.

**Acceptance Criteria:**
1. Given an authenticated Fleet Manager, when they click "Create Work Order", then a form must appear with equipment selector, service type dropdown, priority dropdown, description textarea, date picker, and technician assignment dropdown.
2. Given valid work order data is submitted, when the form is saved, then the work order must be created with status "Open" and a unique work order number must be generated.
3. Given a work order is created and assigned to a technician, when the creation succeeds, then the assigned technician must receive an in-app notification and email notification.
4. Given an Operator user, when they attempt to create a work order, then they must be denied access.
5. Given the equipment selector is displayed, when the user types to search, then equipment must be filtered by name or serial number with autocomplete suggestions.

## L2-008: Work Order Lifecycle Tracking
**Traces to:** L1-003

Work orders must progress through defined statuses: Open → In Progress → On Hold → Completed → Closed. Each status transition must be recorded with timestamp, user, and optional notes. The system must display a work order detail view showing current status, full history, and allow authorized users to update status.

**Acceptance Criteria:**
1. Given a work order with status "Open", when a Technician clicks "Start Work", then the status must change to "In Progress" and a history entry must be recorded with timestamp and user.
2. Given a work order with status "In Progress", when a Technician clicks "Place On Hold" and provides a reason, then the status must change to "On Hold" with the reason recorded.
3. Given a work order with status "In Progress", when a Technician clicks "Complete" and enters completion notes, then the status must change to "Completed".
4. Given a work order with status "Completed", when a Fleet Manager clicks "Close", then the status must change to "Closed" and the work order must become read-only.
5. Given any status transition, when the transition occurs, then the work order history must record the previous status, new status, timestamp, user name, and any notes.

## L2-009: Service Scheduling Calendar
**Traces to:** L1-003

The system must provide a calendar view (day/week/month) for visualizing scheduled service appointments and work orders. Users must be able to drag-and-drop to reschedule and click to view details. The calendar must use Kendo UI Scheduler component.

**Acceptance Criteria:**
1. Given an authenticated Fleet Manager, when they navigate to the service calendar, then a Kendo UI Scheduler must display work orders and service appointments in a calendar format with day, week, and month view options.
2. Given work orders exist for the current week, when the user selects week view, then each work order must appear as a color-coded block (by priority) at its scheduled date/time.
3. Given a Fleet Manager drags a work order to a new date on the calendar, when the drop completes, then the work order's scheduled date must update and a confirmation toast must appear.
4. Given a user clicks a work order block on the calendar, when the click occurs, then a popup must display work order number, equipment name, service type, priority, and assigned technician with a link to the full detail view.
5. Given viewport width < 768px, when the user views the service calendar, then it must default to day view with a simplified layout.

**Traces to:** L1-004

The system must display a searchable, filterable parts catalog organized by category. Each part listing must show: part number, name, description, price, availability status, and compatible equipment models. Users must be able to filter by category, equipment model compatibility, and availability.

**Acceptance Criteria:**

2. Given the parts catalog is displayed, when the user selects a category from the filter sidebar, then only parts in that category must be shown.
3. Given the parts catalog is displayed, when the user selects an equipment model from the compatibility filter, then only parts compatible with that model must be shown.
4. Given a part with availability "In Stock", when displayed in the catalog, then a green "In Stock" badge must be shown.
5. Given a part with availability "Out of Stock", when displayed in the catalog, then a red "Out of Stock" badge must be shown and the "Add to Cart" button must be disabled.

## L2-011: Parts Cart & Order Submission
**Traces to:** L1-004

The system must provide a shopping cart for parts ordering. Users must be able to add parts with quantities, view cart totals, adjust quantities, remove items, and submit the order. Order submission must generate an order number and send a confirmation notification.

**Acceptance Criteria:**
1. Given a part with "In Stock" availability, when the user clicks "Add to Cart" and specifies quantity 3, then the part must be added to the cart with quantity 3 and the cart icon must update to show the item count.
2. Given items in the cart, when the user navigates to the cart page, then all items must be listed with part number, name, unit price, quantity, and line total, plus an order subtotal.
3. Given items in the cart, when the user changes the quantity of an item to 5, then the line total and order subtotal must recalculate immediately.
4. Given items in the cart, when the user clicks "Remove" on an item, then the item must be removed and totals must update.
5. Given a valid cart with items, when the user clicks "Submit Order", then an order must be created with a unique order number, the cart must be cleared, and a confirmation message with the order number must display.
6. Given an order is submitted, when the submission succeeds, then a confirmation email must be sent to the user.

The system must display equipment telemetry data through interactive Kendo UI charts showing engine hours, fuel consumption, operating temperature, and GPS location over configurable time ranges (24h, 7d, 30d, 90d, custom). Charts must auto-refresh every 60 seconds.

**Acceptance Criteria:**
1. Given an authenticated user viewing equipment telemetry, when the page loads, then four Kendo UI charts must display: engine hours (line chart), fuel consumption (bar chart), operating temperature (line chart), and GPS trail (map).
2. Given the telemetry page is displayed, when the user selects "7 days" from the time range selector, then all charts must update to show data for the last 7 days.
3. Given the telemetry page is displayed, when 60 seconds elapse, then chart data must auto-refresh without full page reload.
4. Given no telemetry data exists for the selected time range, when the charts render, then each chart must display a "No data available for this period" message.
5. Given viewport width < 768px, when viewing telemetry, then charts must stack vertically in full-width single-column layout.

## L2-013: Telemetry Event Ingestion
**Traces to:** L1-005, L1-015

The system must ingest telemetry events from equipment via an Azure Function endpoint. Each event must include: equipment ID, timestamp, event type, and payload (JSON). Events must be validated, stored in SQL Server, and trigger alert evaluation. The ingestion endpoint must handle at least 100 events per second.

**Acceptance Criteria:**
1. Given a valid telemetry event payload with equipment ID, timestamp, event type, and JSON data, when posted to the ingestion endpoint, then the event must be stored in the database and a 202 Accepted response must be returned.
2. Given a telemetry event with a missing required field, when posted to the ingestion endpoint, then a 400 Bad Request response must be returned with validation error details.
3. Given a telemetry event for equipment that does not exist, when posted to the ingestion endpoint, then a 404 Not Found response must be returned.
4. Given a telemetry event with a temperature reading above the equipment's critical threshold, when the event is processed, then a critical alert must be generated.
5. Given 100 concurrent telemetry events, when posted to the ingestion endpoint, then all must be processed within 5 seconds with no data loss.

## L2-014: AI Predictive Maintenance
**Traces to:** L1-006

The system must analyze historical telemetry data and service records to generate predictive maintenance recommendations. Recommendations must include: predicted failure component, confidence score, recommended action, and suggested timeframe. Recommendations must be displayed on the equipment detail page and in a dedicated AI Insights dashboard.

**Acceptance Criteria:**
1. Given an equipment with sufficient historical telemetry data (≥30 days), when the AI analysis runs, then predictive maintenance recommendations must be generated and stored.
2. Given a predictive recommendation exists for an equipment, when the user views the equipment detail page, then an "AI Insights" section must display the recommendation with component name, confidence percentage, recommended action, and suggested timeframe.
3. Given multiple recommendations exist across the fleet, when the user navigates to the AI Insights dashboard, then all recommendations must be listed in a sortable grid, ordered by confidence score descending.
4. Given a recommendation with confidence ≥ 80%, when displayed, then it must be highlighted with a red "High Priority" badge.
5. Given a recommendation with confidence < 50%, when displayed, then it must be shown with a gray "Low Confidence" badge.

## L2-015: AI Natural Language Parts Search
**Traces to:** L1-006

The system must provide a natural language search bar in the parts catalog that interprets user queries in plain English (e.g., "hydraulic filter for CAT 320") and returns relevant parts ranked by relevance. The search must handle equipment model references, part category terms, and common misspellings.

**Acceptance Criteria:**
1. Given the parts catalog search bar, when the user types "hydraulic filter for CAT 320", then parts matching hydraulic filters compatible with CAT 320 must be returned, ranked by relevance.
2. Given the user types a query with a common misspelling (e.g., "hydralic"), when the search executes, then it must still return relevant hydraulic parts with a "Did you mean: hydraulic?" suggestion.
3. Given the user types a vague query (e.g., "engine parts"), when the search executes, then results must be returned grouped by subcategory with faceted refinement options.
4. Given the AI search returns results, when displayed, then each result must show relevance score and the matched criteria highlighted.

## L2-016: Anomaly Detection Alerts
**Traces to:** L1-006

The system must continuously analyze incoming telemetry data for anomalous patterns (unusual temperature spikes, abnormal fuel consumption, irregular operating hours). When an anomaly is detected, the system must create an alert and notify the equipment's assigned Fleet Manager.

**Acceptance Criteria:**
1. Given a temperature reading that deviates more than 2 standard deviations from the equipment's 30-day rolling average, when the reading is processed, then an anomaly alert must be created with type "Temperature Anomaly".
2. Given fuel consumption increases by more than 30% compared to the 7-day rolling average without a corresponding increase in operating hours, when detected, then an anomaly alert must be created with type "Fuel Anomaly".
3. Given an anomaly alert is created, when the alert is persisted, then the equipment's assigned Fleet Manager must receive an in-app notification and email.
4. Given the AI Insights dashboard, when an anomaly alert exists, then it must be displayed with the anomaly type, affected equipment, detection timestamp, and severity level.

## L2-018: Organization Multi-Tenancy
**Traces to:** L1-007

The system must ensure complete data isolation between customer organizations. Users must only see equipment, work orders, parts orders, and telemetry data belonging to their own organization. Claims-based authorization must enforce tenant boundaries at the API level.

**Acceptance Criteria:**
1. Given User A belongs to Organization X and User B belongs to Organization Y, when User A requests equipment list, then only Organization X's equipment must be returned.
2. Given User A attempts to access an equipment ID belonging to Organization Y via direct API call, then a 404 Not Found response must be returned (not 403, to prevent information leakage).
3. Given all API endpoints, when a request is made, then the organization claim from the JWT token must be validated and used to filter all database queries.
4. Given an Admin for Organization X, when they view the user list, then only users belonging to Organization X must be visible.

## L2-023: Entra ID Authentication Flow
**Traces to:** L1-010

The system must authenticate users via Microsoft Entra ID using the OAuth2 Authorization Code flow with PKCE. Upon successful authentication, the system must receive an ID token and access token, extract user claims (name, email, roles, organization ID), and establish a session.

**Acceptance Criteria:**
1. Given an unauthenticated user, when they access any protected page, then they must be redirected to the Microsoft Entra ID login page.
2. Given valid Entra ID credentials, when the user completes authentication, then they must be redirected back to the application with a valid session and JWT access token.
3. Given a JWT access token, when it is decoded, then it must contain claims for: user ID, email, display name, roles (array), and organization ID.
4. Given a session that has been idle for 30 minutes, when the user makes a request, then they must be redirected to re-authenticate.
5. Given the user clicks "Sign Out", when the sign-out completes, then the session must be destroyed, tokens must be cleared, and the user must be redirected to the login page.

4. Given an authenticated Technician, when they view a work order assigned to them, then they must be able to update the status but not delete the work order.
5. Given an authenticated Fleet Manager, when they access User Management, then they must receive a 403 Forbidden response.

## L2-025: RESTful API Design
**Traces to:** L1-011

The system must expose RESTful APIs following standard conventions: proper HTTP methods (GET, POST, PUT, DELETE), consistent JSON response envelopes, pagination for list endpoints, appropriate HTTP status codes, and API versioning via URL path (e.g., /api/v1/). All APIs must be documented with OpenAPI/Swagger.

**Acceptance Criteria:**
1. Given the API base URL, when a GET request is made to /api/v1/equipment, then a JSON response must be returned with a paginated list of equipment including metadata (total count, page size, current page).
2. Given a POST request to /api/v1/equipment with a valid JSON body, when the request is processed, then the equipment must be created and a 201 Created response must be returned with the created resource and a Location header.
3. Given a GET request to /api/v1/equipment/{id} for a non-existent ID, when processed, then a 404 Not Found response must be returned with a standard error envelope.
4. Given the API documentation endpoint, when accessed, then a Swagger UI must be rendered with complete documentation for all endpoints.
5. Given any API request without a valid JWT bearer token, when processed, then a 401 Unauthorized response must be returned.

## L2-026: Legacy XML Data Import
**Traces to:** L1-011

The system must support importing equipment data from legacy XML feeds. The import process must validate the XML against a defined schema, transform the data to the internal model, handle duplicates by matching on serial number, and report import results (created, updated, skipped, errors).

**Acceptance Criteria:**
1. Given a valid XML file conforming to the legacy equipment schema, when uploaded via the import endpoint, then equipment records must be created or updated in the database.
2. Given an XML file with an equipment record whose serial number already exists, when processed, then the existing record must be updated rather than duplicated.
3. Given an XML file with invalid data (e.g., missing required elements), when processed, then the import must skip invalid records, process valid ones, and return a summary with error details for each failed record.
4. Given an XML file exceeding 10MB, when uploaded, then a 413 Payload Too Large response must be returned.

## L2-029: Input Validation & OWASP Protections
**Traces to:** L1-013

The system must validate all user inputs on both client and server side. The system must implement protections against OWASP Top 10 vulnerabilities including: SQL injection (parameterized queries), XSS (output encoding), CSRF (anti-forgery tokens), broken authentication (token validation), and security misconfiguration.

**Acceptance Criteria:**
1. Given a POST request with a string field containing SQL injection payload (e.g., "'; DROP TABLE--"), when processed, then the input must be rejected with a 400 Bad Request or safely parameterized, and no SQL injection must occur.
2. Given a field value containing a script tag (e.g., "<script>alert('xss')</script>"), when rendered in the UI, then the content must be HTML-encoded and not executed.
3. Given a state-changing request (POST/PUT/DELETE), when submitted without a valid anti-forgery token, then the request must be rejected with a 400 Bad Request response.
4. Given all API endpoints, when called, then response headers must include: X-Content-Type-Options: nosniff, X-Frame-Options: DENY, and Content-Security-Policy with appropriate directives.
5. Given rate limiting is configured, when more than 100 requests per minute are made from the same IP to any API endpoint, then subsequent requests must receive a 429 Too Many Requests response.

## L2-030: API Rate Limiting & Security Headers
**Traces to:** L1-013

The system must implement rate limiting at the API gateway level and include security headers on all responses. Rate limits must be configurable per endpoint and per client tier.

**Acceptance Criteria:**
1. Given rate limiting is configured for 100 requests/minute on the equipment endpoint, when the 101st request arrives within a minute, then a 429 Too Many Requests response must be returned with a Retry-After header.
2. Given any API response, when headers are inspected, then X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, and Content-Security-Policy headers must be present.
3. Given the rate limit is exceeded, when the response is returned, then it must include X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers.

## L2-031: Responsive Navigation
**Traces to:** L1-014

The system must provide a responsive navigation structure. On large viewports (≥992px), a persistent left sidebar must display navigation links with icons and labels. On medium viewports (≥768px, <992px), the sidebar must collapse to icons only. On small and extra-small viewports (<768px), the sidebar must be hidden and replaced with a hamburger menu that opens a full-screen overlay navigation.

**Acceptance Criteria:**
1. Given viewport width ≥ 992px, when any page loads, then a persistent left sidebar must display navigation links with icons and text labels for: Dashboard, Equipment, Service, Parts, Telemetry, AI Insights, Reports, and Admin (if authorized).
2. Given viewport width ≥ 768px and < 992px, when any page loads, then the sidebar must show only icons with tooltips on hover.
3. Given viewport width < 768px, when any page loads, then the sidebar must be hidden and a hamburger icon must appear in the top navigation bar.
4. Given viewport width < 768px and the user clicks the hamburger icon, then a full-screen overlay navigation must appear with all menu items.
5. Given the overlay navigation is open, when the user clicks a menu item, then they must be navigated to that page and the overlay must close.

## L2-032: Responsive Data Tables
**Traces to:** L1-014

All data tables in the system must adapt to viewport size. On large viewports, full column sets must display. On medium viewports, secondary columns must be hidden with an expand row option. On small/extra-small viewports, tables must transform to a card-based layout.

**Acceptance Criteria:**
1. Given viewport width ≥ 992px, when a data table is displayed, then all columns must be visible.
2. Given viewport width ≥ 768px and < 992px, when a data table is displayed, then secondary columns (e.g., last service date, location) must be hidden with a "+" expand button to reveal them inline.
3. Given viewport width < 768px, when a data table is displayed, then data must render as stacked cards with key-value pairs, one card per record.
4. Given the card layout is displayed, when the user taps a card, then it must expand to show all fields.

## L2-033: Azure Function Telemetry Processing
**Traces to:** L1-015

The system must use Azure Functions to process telemetry events asynchronously. Functions must: validate incoming events, enrich with equipment metadata, evaluate alert thresholds, persist to SQL Server, and publish events for downstream consumers (anomaly detection, real-time dashboards).

**Acceptance Criteria:**
1. Given a telemetry event is received by the Azure Function, when processed, then it must be validated, enriched with equipment make/model/organization, and persisted to the telemetry table.
2. Given a telemetry event with a reading exceeding a configured threshold, when processed, then the function must create an alert record and trigger the notification pipeline.
3. Given the Azure Function processes a batch of 50 telemetry events, when executed, then all events must be processed within 10 seconds.
4. Given the Azure Function encounters a transient database error, when the error occurs, then it must retry up to 3 times with exponential backoff before dead-lettering the event.

## L2-034: Logic App Service Workflows
**Traces to:** L1-015

The system must use Azure Logic Apps to orchestrate service workflows including: automated service reminder sequences (7 days, 3 days, 1 day before), work order escalation (auto-escalate if overdue by 48 hours), and parts order fulfillment status updates from external systems.

**Acceptance Criteria:**
1. Given an equipment has a scheduled service in 7 days, when the Logic App daily trigger fires, then a "7-day service reminder" notification must be sent to the Fleet Manager.
2. Given a work order has been in "Open" status for more than 48 hours, when the Logic App hourly trigger fires, then the work order priority must be escalated by one level and a notification must be sent to the Fleet Manager.
3. Given a parts order status update is received from an external fulfillment system, when the Logic App processes it, then the order status must be updated in the database and the ordering user must be notified.
